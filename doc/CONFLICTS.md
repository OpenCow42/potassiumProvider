# Current Conflict Resolution

This document describes the conflict behavior in `potassiumProvider`.

The provider is still mostly server-authoritative for final metadata: create,
replace, move, rename, trash, and delete requests are sent to kDrive, and server
responses are returned to File Provider. The high-risk guardrail now in place is
that stale local mutations are detected before the server is changed.

The listing cache introduced by `KDriveSnapshotSQLiteStore` is a metadata cache.
It helps enumerate folders and diff server changes, but it is not a full sync
database or durable pending-operation journal.

## Current Principles

The implementation follows these practical rules:

1. The server is the source of truth for item identity and final metadata.
2. Local mutations fetch latest metadata before changing existing items.
3. File Provider base versions are compared with latest remote versions.
4. Stale content replacement preserves both by uploading a renamed conflict copy.
5. Stale rename, move, trash, and permanent delete are blocked before server
   mutation.
6. Later enumeration or change sync updates the SQLite snapshot from kDrive
   state.

This is a focused conflict-safety pass, not the full sync-database redesign.

## Version Checks

`FileProviderItem` derives versions from kDrive metadata:

- `contentVersion`: `modifiedAt.timeIntervalSince1970`
- `metadataVersion`: `id`, `updatedAt`, `name`, and `parentID`

Before mutating an existing item, the extension fetches latest metadata with
`item(driveID:fileID:)` and compares the relevant base version:

- content replace checks content version
- rename and move check metadata version
- trash and permanent delete check content and metadata versions

These are timestamp-based conflict tokens. A future implementation should prefer
a kDrive revision, etag, checksum, or version ID if one is available.

## Create Conflicts

### File Create Versus Existing File With Same Name

For new files, `createItem(...)` uploads with
`KDriveUploadConflictStrategy.version` and returns the server-created item.

Current behavior:

- The extension does not preflight the parent listing for an existing sibling
  with the same name.
- The extension does not return `NSFileProviderError.filenameCollision`.
- kDrive receives `conflict=version` and decides the final result.

This remains server-authoritative. It relies on kDrive version behavior rather
than provider-generated preserve-both names.

### File Or Folder Create Versus Existing Folder/File

Folder creation still calls `createDirectory(...)` without an explicit local
collision preflight. File creation still uses `conflict=version`.

This is not yet fully conflict-safe. A later pass should define folder collision
policy and file-versus-folder behavior explicitly.

## Content Modify Conflicts

When `modifyItem(...)` includes `.contents`:

1. Fetch latest remote metadata.
2. Compare File Provider `baseVersion.contentVersion` with the latest remote
   content version.
3. If versions match, call `replaceFile(...)` on the original item.
4. If versions differ, stage the local bytes in app-group `ConflictStaging`.
5. Upload those bytes to the latest remote parent with `conflict=rename`.
6. Return the conflict-copy item and leave the original untouched.

Conflict names are generated by `KDriveConflictFilename`, for example:

```text
Report (conflict - Alice's MacBook - 2026-07-05 17.58.00).pdf
```

The name preserves the extension, uses the current device name when available,
and is deterministic when the device name, date, and time zone are injected in
tests.

Remaining limitation:

The staged conflict file is not tracked by a durable pending-operation table.
It is removed after successful conflict upload and retained if upload fails, but
there is not yet an automatic retry/recovery workflow.

## Metadata Modify Conflicts

Rename and move now fetch latest metadata before mutating:

- matching metadata base version: send the rename or move to kDrive
- stale metadata base version: do not mutate the server; return a recoverable
  File Provider error

Move still requests `conflict=rename` from kDrive for name collisions in the
destination parent.

The provider does not yet preflight sibling names or return
`NSFileProviderError.filenameCollision` for local rename collisions.

## Trash And Delete Conflicts

Trash and permanent delete now fetch latest metadata before mutating:

- matching content and metadata base versions: send the trash/delete request
- stale base version: do not mutate the server; return a recoverable File
  Provider error

On the current iOS File Provider target, stale destructive mutations are returned
as `.cannotSynchronize` with a recovery suggestion to refresh and retry. This is
the platform-compatible form of the intended "version no longer available"
behavior.

Delete-delete remains server-defined. If the item no longer exists remotely,
kDrive's response is mapped through `providerError(...)`.

## Listing And Snapshot Conflicts

Normal folders use advanced listing for enumeration and change sync. Listing is
now fail-closed:

- repeated cursors throw instead of committing partial listings
- `hasMore == true` without a continuation cursor throws
- unknown advanced actions throw
- update actions without item metadata throw
- delete actions may omit metadata and still delete locally

Enumeration and change paths save snapshots with
`KDriveSnapshotSaveCondition.missing` or
`.matching(anchor:serverCursor:)`. If another enumerator has already advanced the
snapshot, the stale writer receives `KDriveSnapshotStoreError.staleSnapshot` and
does not overwrite newer cursor state.

Invalid advanced change payloads return `.syncAnchorExpired`; invalid legacy
listing loops return `.cannotSynchronize`.

## Retry And Ambiguous Success Conflicts

There is still no durable pending-operation table and no provider-layer
idempotency key. If a create or mutation succeeds on the server but the extension
times out before File Provider receives success, retry behavior is still mostly
controlled by kDrive.

This can create duplicates, unexpected versions, or mapped errors. It is usually
safer than silently dropping data, but it remains a major reason to add a
sync-operation journal.

## Current Data-Loss Risk Summary

Lower risk today:

- Stale content replace, because the provider creates a renamed conflict copy.
- Stale rename, move, trash, and delete, because the provider blocks the server
  mutation before overwriting newer remote state.
- Cursor races, because guarded snapshot saves prevent stale cache writers from
  regressing stored cursor state.
- Malformed listing pages, because cursor/action anomalies fail closed.

Medium risk today:

- File create-create, because files use `conflict=version` and depend on kDrive
  version retention.
- Rename-to-existing-name, because sibling collision behavior is still delegated
  to kDrive unless the stale metadata check catches a concurrent change.
- Operation timeout after server success, because there is no idempotency or
  pending-operation journal.

Higher risk today:

- Folder create conflicts, because no explicit conflict policy is passed.
- Parent deleted while child is created or modified, because there is no recovered
  folder policy.
- Failed conflict upload recovery, because staged bytes are retained but not yet
  surfaced through an automatic retry UI.

## Recommended Safe Direction

For a data-loss-averse provider, future work should still:

1. Add a SQLite pending-operation journal for creates, modifies, deletes, and
   retries.
2. Store local upload bytes durably before every network send, not only stale
   content conflict copies.
3. Prefer `conflict=rename` for file creates when preserve-both is desired.
4. Pass an explicit directory conflict policy when creating folders.
5. Return `NSFileProviderError.filenameCollision` for local name collisions that
   File Provider can safely rename.
6. Move from timestamp-derived versions to authoritative kDrive revision tokens
   if available.
7. Treat parent-deleted scenarios as recovery cases with staged child contents.
8. Reconcile by stable item ID where possible, and by parent plus filename only
   before the server assigns an ID.

## Bottom Line

The three high-risk guardrails are now addressed for snapshot races, malformed
listing state, and stale existing-item mutations. The provider is still not a
full conflict-safe sync engine until pending operations, broader staged uploads,
explicit create collision handling, and stronger server version tokens are in
place.
